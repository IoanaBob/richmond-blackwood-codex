# RB Presentation Drafting

Use this skill for Richmond Blackwood decks, pitch packets, and Figma/Slides creation.

## Core Workflow

1. Gather product, audience, and proof context before drafting. Register source context separately.
2. Draft a slide-by-slide packet before creating or changing the live deck.
3. Map every slide to an exact source template slide, page, or node before Figma work.
4. Create in Figma by copying the matching template slide and editing existing elements.
5. Verify the live deck with screenshots or Figma readback before closeout.

## Template Rules

- Treat the template as the design authority.
- Copy the matching template slide first. Edit text, existing shapes, and existing component slots in place.
- Preserve template typography, color treatment, and component styling.
- Preserve spacing and layout unless the user explicitly asks for a redesign.
- Do not recreate template graphic elements from scratch when the template already contains the needed structure. This includes cards, tables, and rails. Treat logos and icons as existing assets.
- For table or card emphasis, duplicate or reposition the existing template highlight. Adjust opacity, stroke, or placement before introducing a new visual style.

## Slide Writing Rules

- Eyebrows should usually be known deck/navigation labels. Common labels include:
  - `PROBLEM`
  - `WHY IT FAILED`
  - `SOLUTION`
  - `HOW IT WORKS`
  - `USE CASES`
  - `PACKAGES`
  - `NEXT STEP`
- Match the template's eyebrow structure and capitalization. Do not add eyebrows to cover or closing slides if the template omits them.
- Titles must carry the point of the slide. Avoid vague labels that make the subtitle do all the work.
- Do not repeat the same key phrase across the title, subtitle, and body.
- Keep visible body copy to two sentences or fewer.
- Do not enumerate more than three items in one sentence or clause. Use bullets, tables, or appendix notes for longer lists.
- Avoid mechanical contrast sentences in pitch copy. State the useful claim directly.

## Evidence And Appendix

- Use public sources, Notion context, or repo context for factual claims.
- Put calculation detail, source notes, and caveats into the slide appendix or speaker notes. Add supporting examples there too when needed.
- Make appendix notes slide-specific when possible.

## Visual Direction

- Professional-services decks should feel serious, concise, and evidence-backed.
- Prefer template-native tables, cards, and diagrams over newly drawn graphics.
- Use icons only where the template already expects icon-like content or the user asks for them.
- Do not make operational slides look playful or childlike.
- Data slides should show a few strong metrics with source notes, not dense paragraphs.
